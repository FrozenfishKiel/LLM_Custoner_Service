# 电商智能客服系统开发交接文档

## 1. 交接目标

接手者需要基于已经定型的PRD和技术方案，将现有单机课程Demo改造成可公网注册、登录和稳定使用的电商智能客服系统，并完成测试、部署、监控、备份、量化和上线验收。

本次工作不是重新讨论产品方向，也不是把项目扩展成商城或多商家SaaS。需求边界已经固定，开发必须以文档为准。

## 3. 必读顺序

接手后必须按顺序完整阅读：

1. `docs/HANDOFF.md`
2. `docs/REQUIREMENTS_DECISIONS.md`
3. `docs/PRD.md`
4. `docs/TECHNICAL_DESIGN.md`
5. 当前代码、配置和启动脚本

文档职责：

- `REQUIREMENTS_DECISIONS.md`：记录讨论中已经确认的产品决策。
- `PRD.md`：定义用户、功能、非功能要求和上线验收标准。
- `TECHNICAL_DESIGN.md`：定义模块、接口、数据、Redis、部署、测试和文件改动范围。
- `HANDOFF.md`：说明当前状态、工作目录、开工规则和风险。

出现冲突时，优先级为：用户最新明确指令 > 需求决策记录 > PRD > 技术方案 > 现有代码行为。不得因为现有代码实现简单就降低需求标准。

## 4. 已确认产品边界

必须实现：

- 消费者邮箱注册、邮箱验证和密码登录
- Session、退出、修改密码、忘记密码、重置密码和注销账号
- 注册后为账号初始化少量独立模拟业务数据
- 登录用户只能访问自身的订单、物流、地址和售后数据
- 订单查询、物流查询、修改收货地址、取消订单和售后申请
- 电商客服范围内的知识问答和边界兜底
- 最小管理员账号、账号状态管理、模拟数据重置、指标和审计查看
- Redis Session、验证码、限流和Tracker
- MySQL、Neo4j、DeepSeek和SMTP集成
- 正式网页、HTTPS、监控、告警、备份、恢复和回滚
- 自动化测试、LLM评测、压测、安全检查和量化报告

明确禁止：

- 多商家、多租户和商家管理
- 商品、购物车、商城和下单
- 支付、充值、订阅、套餐、账单和发票
- 真实商家数据接入
- 微信、企业微信、抖音等外部渠道
- 人工客服工作台和工单系统
- 为展示技术数量引入消息队列、微服务或Kubernetes

## 6. 当前代码不是目标状态

当前代码可以运行，但仍然是教学Demo。已知需要按技术方案处理的问题包括：

- API没有正式消费者认证和管理员鉴权。
- 客户端可以传入任意 `sender_id`，存在会话和用户越权风险。
- 原有Flow允许切换任意 `user_id`，生产环境必须禁用。
- CORS默认开放，调试页、API文档和Tracker接口可能暴露。
- 当前Tracker主要使用内存配置，尚未实现Redis TrackerStore。
- 数据库连接和辅助脚本中仍可能存在固定账号密码。
- 当前没有消费者注册、邮箱验证、密码重置和注销流程。
- 当前没有账号与业务 `user_id` 的可信绑定。
- 业务写操作需要补充归属校验、幂等、审计和一致性测试。
- `EnterpriseSearchPolicy`的闲聊提示曾表现为通用AI助手，必须按PRD限制为电商客服身份。
- 当前缺少正式客服页面、管理员页面、自动化测试、指标、告警和生产部署编排。
- 当前目录尚未建立可靠的Git版本基线。

不要把这些问题理解为允许重写整个框架。现有Agent、LangGraph、Flow、Action和GraphRAG必须优先保留，并在技术方案指定的切口上改造。

## 7. 密钥和敏感信息

现有 `ecs_demo/.env` 中包含真实模型密钥和本地数据库凭证。

强制要求：

- 不得在聊天、日志、测试输出、提交记录和交接文档中打印密钥值。
- 不得提交 `.env`。
- 开发前创建 `.env.example`，只保留变量名和安全占位符。
- 生产上线前撤销并轮换已经暴露过的DeepSeek Key。
- MySQL使用非root应用账号，Neo4j和Redis只允许内部网络访问。

## 8. 开工规则

接手者不得直接从某个接口开始编码。开工顺序固定为：

1. 在`D:`标准工作副本核对文档与代码。
2. 建立Git基线并确认 `.gitignore` 不会提交密钥、模型缓存、数据库卷和运行日志。
3. 根据技术方案编写可执行实施计划，逐项关联PRD编号和验证命令。
4. 先完成账号、身份、数据归属和数据库迁移设计，再修改业务Action。
5. 每个实施任务使用测试驱动或至少先定义可验证验收条件。
6. 每完成一个模块立即运行对应测试，不把全部验证推迟到最后。
7. 只有完整完成定义通过后，才能声称项目完成。

如果实施中发现技术方案遗漏：

- 先说明遗漏对应哪个PRD需求。
- 给出最小修改方案和影响范围。
- 更新技术方案后再实现。
- 不得顺手加入PRD明确排除的功能。

## 9. 实施优先顺序

实施计划可以细化，但依赖顺序不得颠倒：

```text
Git与配置安全
  -> 数据库迁移和账号模型
  -> Redis基础设施和Session
  -> 邮箱注册、验证、登录和密码流程
  -> 账号与业务user_id绑定
  -> 模拟数据初始化
  -> Chat认证和Tracker迁移
  -> Action归属、事务、幂等和审计
  -> 正式客服前端
  -> 管理员最小功能
  -> 日志、Prometheus、Grafana和告警
  -> Docker、Nginx、备份和发布
  -> 自动化测试、LLM评测、压测和故障演练
  -> 云端部署、备案前置和连续运行
```

这是施工依赖顺序，不是允许交付缩水版本的产品阶段。所有内容完成后才算当前版本结束。

## 10. 外部依赖与用户操作

以下事项可以由AI准备技术配置和操作说明，但必须由用户本人完成账号实名或审批：

- 云服务器购买和云账号实名认证
- 域名购买和实名认证
- ICP备案、短信核验和可能的公安联网备案
- SMTP邮箱或事务邮件服务账号配置
- DeepSeek生产Key创建和费用充值
- 最终隐私政策、用户协议及合规确认

不得用本地沙箱成功替代这些正式上线前置条件。

## 11. 验证要求

至少需要输出并保留：

- 单元测试报告
- MySQL、Redis、Neo4j和SMTP集成测试报告
- 注册、登录、数据隔离和五类客服流程的浏览器E2E报告
- LLM意图、槽位、Flow和边界评测报告
- 20并发客服会话压测报告
- 安全检查和依赖扫描结果
- MySQL及Neo4j备份恢复演练记录
- 发布和回滚演练记录
- Prometheus/Grafana监控截图或导出配置
- 连续运行期间的真实指标和问题修复记录

目标值不是成果值。简历只能使用实际测试和运行得到的数据。

## 12. 完成交接判定

接手者在开始开发前应当能够明确回答：

- 产品为什么需要账号，以及账号如何绑定模拟订单。
- MySQL、Redis、Neo4j和DeepSeek分别负责什么。
- 为什么用户身份不能从LLM、消息或请求体获得。
- 哪些现有模块保留，哪些模块新增，哪些功能明确禁止。
- 每个PRD编号通过什么测试或运行证据验收。

若无法回答，继续阅读PRD、技术方案和现有代码，不要开始实现。
