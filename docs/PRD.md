# 电商智能客服系统 PRD

## 1. 文档信息

| 项目 | 内容 |
| --- | --- |
| 产品名称 | 电商智能客服系统 |
| 文档版本 | v2.0 |
| 文档状态 | 已定型，可作为技术设计和开发依据 |
| 产品形态 | 单商家、非收费、可公网访问的电商智能客服系统 |
| 业务数据 | 为注册用户初始化的独立模拟电商数据 |
| 目标用户 | 希望体验订单、物流和售后客服流程的注册用户 |

本 PRD 与 `docs/REQUIREMENTS_DECISIONS.md` 共同构成当前版本的需求基线。后续新增需求必须先修改需求基线，不得在开发过程中直接追加。

## 2. 产品背景与目标

现有项目是基于大语言模型、Flow、Action、MySQL 和 Neo4j/GraphRAG 构建的电商智能客服 Demo，已经具备订单查询、物流查询、修改收货信息、取消订单和售后申请等业务能力，但缺少正式账号、数据隔离、生产部署、监控、测试和上线运营能力。

本版本的目标是把现有 Demo 改造成一个可以公开注册、登录和稳定使用的线上系统。用户完成邮箱验证并登录后，系统为其准备独立的模拟电商业务数据，用户可以通过自然语言完成原项目已有的客服业务流程。

本产品不建设商城，也不产生真实交易。模拟数据只用于验证和体验智能客服能力。

## 3. 产品原则

- 客服回复中的订单、物流、地址和售后事实必须来自业务数据库，不得由模型编造。
- 每个注册账号只能访问和修改与自身绑定的模拟业务数据。
- 用户登录后应当可以直接完成核心客服流程，不要求额外配置。
- 账号体系服务于身份识别和数据隔离，不扩展为社交、会员或支付体系。
- 产品功能、运行指标和简历成果必须由真实实现和测量数据支撑。
- 不为追求技术数量引入与当前上线目标无关的业务能力。

## 4. 用户角色

### 4.1 消费者

消费者可以自行注册账号、验证邮箱、登录系统，并使用智能客服查询或操作自己的模拟订单、物流、地址和售后数据。

### 4.2 系统管理员

系统管理员负责必要的线上维护，包括账号状态管理、指定用户模拟数据重置、运行指标查看和审计记录查看。

管理员不是商家，不负责商品、交易、支付、客服排班或商家配置。

## 5. 产品范围

### 5.1 必须实现

- 邮箱注册、邮箱验证和密码登录
- 保持登录状态、退出登录和修改密码
- 忘记密码和邮件重置密码
- 账号注销
- 登录后才能访问客服功能
- 注册完成后初始化独立模拟业务数据
- 网页客服对话界面
- 多轮对话和上下文保持
- 订单列表及订单详情查询
- 物流信息查询
- 修改收货信息
- 取消订单
- 退款、退货和换货申请
- 电商客服范围内的知识问答和边界兜底
- 管理员账号管理、模拟数据重置、指标和审计查看
- 云端部署、域名、HTTPS和必要备案
- 日志、监控、告警、备份和恢复
- 自动化测试、LLM评测和量化报告

### 5.2 明确不实现

- 多商家 SaaS 和租户管理
- 商家注册、商家后台和商家自助配置
- 商品展示、搜索、购物车和下单
- 真实商品交易和真实订单接入
- 支付宝、微信支付、充值、订阅、套餐和账单
- 发票、结算和财务系统
- 微信公众号、企业微信、抖音等外部客服渠道
- 人工客服工作台、排班、绩效和完整工单系统
- 移动端原生 App

## 6. 核心用户流程

### 6.1 注册与首次使用

```text
访问注册页
  -> 输入邮箱和密码
  -> 接收并完成邮箱验证
  -> 系统创建账号和业务用户
  -> 系统初始化独立模拟地址、订单、物流和售后场景
  -> 用户登录
  -> 进入客服页面
```

模拟数据应覆盖待发货、已发货、已签收等代表性状态，保证用户能够体验查询、修改地址、取消订单和售后申请。

### 6.2 客服主链路

```text
登录用户输入问题
  -> 系统从登录态确定当前用户
  -> 大模型识别意图并提取槽位
  -> Flow推进多轮业务流程
  -> Action校验数据归属和业务状态
  -> 查询或修改MySQL业务数据
  -> 必要时通过GraphRAG检索客服知识
  -> 返回客服回复并保存对话状态
```

用户身份只能从登录态获得，不能通过消息、URL或请求体切换为任意 `user_id`。

### 6.3 账号注销

```text
用户进入账号设置
  -> 再次验证密码
  -> 确认注销
  -> 撤销全部登录状态
  -> 删除账号、用户资料及关联模拟业务数据
  -> 历史指标仅保留不可反查身份的聚合结果
```

## 7. 功能需求

### 7.1 账号与身份

| 编号 | 需求 | 验收标准 |
| --- | --- | --- |
| A-01 | 邮箱注册 | 未注册邮箱可以创建待验证账号，重复邮箱被拒绝 |
| A-02 | 邮箱验证 | 有效验证码或验证链接可以激活账号，过期或重复使用被拒绝 |
| A-03 | 密码登录 | 已验证账号使用正确密码可以登录，错误密码不泄露账号存在状态 |
| A-04 | 登录保持 | 刷新页面后登录状态保持，过期或撤销后需要重新登录 |
| A-05 | 退出登录 | 退出后当前登录状态立即失效 |
| A-06 | 修改密码 | 用户验证原密码后可以修改，新密码生效后撤销其他登录状态 |
| A-07 | 忘记密码 | 用户可以通过邮箱获得一次性重置凭证 |
| A-08 | 重置密码 | 有效凭证可以设置新密码，凭证使用后立即失效 |
| A-09 | 注销账号 | 再次验证后删除账号及关联模拟业务数据 |
| A-10 | 登录保护 | 未登录用户不能访问客服、用户资料和业务接口 |

### 7.2 模拟数据初始化

| 编号 | 需求 | 验收标准 |
| --- | --- | --- |
| D-01 | 创建业务用户 | 邮箱验证完成后创建唯一业务用户并绑定账号 |
| D-02 | 初始化地址 | 为用户创建可用于修改收货信息流程的模拟地址 |
| D-03 | 初始化订单 | 为用户创建覆盖主要客服场景的少量模拟订单 |
| D-04 | 初始化物流 | 为已发货和已签收订单创建匹配的物流记录 |
| D-05 | 数据隔离 | 任何查询和写操作都只能处理当前登录用户的数据 |
| D-06 | 初始化幂等 | 重复验证或重试不能重复创建同一批模拟数据 |

### 7.3 客服前端

| 编号 | 需求 | 验收标准 |
| --- | --- | --- |
| U-01 | 客服对话页 | 登录用户可以输入消息并看到结构化回复 |
| U-02 | 多轮对话 | 用户可以继续补充订单号、地址和售后信息 |
| U-03 | 业务按钮 | 系统能够展示订单选择和确认按钮并处理点击结果 |
| U-04 | 加载状态 | 模型处理中页面有明确且不重复提交的加载状态 |
| U-05 | 异常状态 | 网络、模型或数据库失败时展示可理解错误 |
| U-06 | 新建会话 | 用户可以清空当前对话状态并开始新会话 |
| U-07 | 账号设置 | 用户可以修改密码、退出和注销账号 |
| U-08 | 调试隔离 | 普通用户无法访问Prompt、Flow、Tracker和内部配置 |

### 7.4 对话理解

| 编号 | 需求 | 验收标准 |
| --- | --- | --- |
| C-01 | 意图识别 | 能区分订单、物流、地址修改、取消和售后场景 |
| C-02 | 槽位提取 | 能提取订单号、地址、售后类型和原因等参数 |
| C-03 | 缺参追问 | 必要槽位缺失时继续追问，不提前执行Action |
| C-04 | 上下文保持 | 多轮对话保持当前Flow和已收集槽位 |
| C-05 | 意图切换 | 用户改口时可以终止或切换当前Flow并清理临时槽位 |
| C-06 | 客服边界 | 非业务问题保持电商客服身份，不表现为无限制通用助手 |
| C-07 | 超时处理 | 模型超时或失败时返回稳定错误，不生成虚假业务结果 |

### 7.5 业务动作

| 编号 | 需求 | 验收标准 |
| --- | --- | --- |
| B-01 | 查询订单 | 只返回当前用户的数据库订单 |
| B-02 | 查询详情 | 返回订单状态、商品、金额和收货信息 |
| B-03 | 查询物流 | 返回当前用户订单对应的物流公司、单号和节点 |
| B-04 | 修改地址 | 状态允许且二次确认后更新收货信息 |
| B-05 | 取消订单 | 状态允许且二次确认后取消订单 |
| B-06 | 售后资格 | 根据订单状态判断是否允许退款、退货或换货 |
| B-07 | 创建售后 | 确认后创建可追踪的售后记录 |
| B-08 | 事务一致性 | 写操作失败时完整回滚 |
| B-09 | 幂等控制 | 重复提交不会重复取消或重复创建售后记录 |
| B-10 | 审计 | 关键写操作记录用户、对象、结果、时间和请求标识 |

### 7.6 管理员

| 编号 | 需求 | 验收标准 |
| --- | --- | --- |
| M-01 | 管理员登录 | 管理员使用独立权限进入受保护管理入口 |
| M-02 | 账号检索 | 可以按邮箱或状态查询账号，不显示密码和密钥 |
| M-03 | 启用与禁用 | 可以禁用或恢复账号，禁用后撤销该账号登录状态 |
| M-04 | 数据重置 | 可以二次确认后重建指定用户的模拟业务数据 |
| M-05 | 指标查看 | 可以查看请求、错误、延迟、Flow和模型调用指标 |
| M-06 | 审计查看 | 可以查看登录、账号管理和关键业务写操作记录 |

## 8. 非功能需求

### 8.1 安全

- 密码必须使用安全密码哈希算法保存，禁止明文和可逆加密。
- 登录状态、邮箱验证码和重置凭证必须具备有效期和撤销能力。
- 注册、登录、验证码、重置密码和客服接口必须限流。
- 用户身份只从服务端登录状态读取。
- 所有订单和售后操作必须验证数据归属。
- 管理员接口必须执行角色鉴权。
- 生产环境只允许正式域名跨域访问。
- Cookie必须使用 HttpOnly、Secure 和合理的 SameSite 策略。
- 调试页面、API文档、Tracker详情和内部指标不得直接暴露公网。
- 日志不得记录密码、完整Session、验证码、模型密钥和未经脱敏的个人信息。

### 8.2 性能与容量

当前版本的初始目标：

- 支持1000个注册账号。
- 支持100个日活账号。
- 支持20个并发客服会话。
- 非模型接口P95响应时间不超过1秒。
- 完整客服请求P95响应时间不超过15秒。
- 月度服务可用性目标为99.5%。

这些是设计和验收目标，不是已经取得的成果。最终能力必须以压测和实际运行结果为准。

### 8.3 可恢复性

- 应用异常退出后能够自动重启。
- MySQL、Redis和Neo4j数据使用持久化存储。
- MySQL和Neo4j具备备份与恢复流程。
- Redis故障不能造成订单和售后业务事实丢失。
- 发布失败时能够回滚到上一可用版本。
- 模型或依赖不可用时不得返回伪造业务结果。

### 8.4 合规与隐私

- 提供隐私政策、用户协议、联系方式和注销入口。
- 明确说明系统使用模拟业务数据，不发生真实交易。
- 说明邮箱、日志和会话数据的用途与保存规则。
- 中国大陆公网部署需完成域名实名认证、HTTPS和适用的ICP备案等手续。
- 生成式AI内容标识、投诉反馈、数据处理和相关备案要求根据实际部署主体进一步核验。

## 9. 量化指标

### 9.1 产品与业务效果

- 注册成功数和邮箱验证成功率
- 登录成功率
- 有效会话数
- 各Flow触发数和完成数
- 业务任务完成率
- 平均完成轮次
- 用户满意度

### 9.2 模型效果

- 意图识别准确率
- 槽位提取准确率
- 兜底率
- 越界回答率
- 模型错误率
- 单次请求Token数和模型成本

### 9.3 系统指标

- 请求量和并发会话数
- P50/P95响应时间
- 4xx和5xx错误率
- 服务可用性
- MySQL、Redis和Neo4j连接异常数
- 发布次数、故障次数和平均恢复时间

指标必须具备固定定义、统计周期和数据来源，不得将目标值或模拟值写成已取得成果。

## 10. 测试要求

- 账号注册、邮箱验证、登录、退出和密码重置测试
- 未登录、越权、禁用账号和管理员权限测试
- 用户间订单、地址、物流和售后隔离测试
- 模拟数据初始化幂等测试
- 五类客服业务端到端测试
- 重复提交、事务回滚和依赖故障测试
- LLM意图、槽位、Flow切换和边界评测
- 接口限流、输入校验和敏感信息泄露测试
- 20并发会话目标的压力测试
- 备份恢复和发布回滚演练

## 11. 上线验收标准

以下条件全部满足后，才能认定本版本完成：

- 用户可以完成邮箱注册、验证、登录、密码管理和注销。
- 每个账号拥有独立模拟业务数据，用户间无法越权访问。
- 五类核心客服业务完成端到端验收。
- 管理员最小功能可以正常使用且权限隔离有效。
- MySQL、Redis、Neo4j和模型服务异常时系统行为符合预期。
- 生产密钥未进入代码、镜像、日志和版本记录。
- 公网服务使用HTTPS，调试接口未暴露。
- 自动化测试、LLM评测、压测和安全检查通过既定门槛。
- 备份恢复和发布回滚完成实际演练。
- 监控和告警能够发现服务及依赖异常。
- 系统连续运行并产生可核验的指标报告和问题修复记录。
- 域名、备案、隐私政策和用户协议等上线前置事项已完成。

## 13. 需求变更规则

当前版本完成前，以下内容不得作为“顺手增加”进入开发：商城、支付、多商家、外部渠道、人工客服工作台和移动App。

任何新需求都必须说明：

1. 对应的用户问题。
2. 是否属于本PRD产品范围。
3. 对数据模型、接口、部署和测试的影响。
4. 是否会改变当前完成时间和验收标准。

未经需求基线修改和重新确认，不进入实施计划。
