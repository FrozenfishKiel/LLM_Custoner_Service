# 需求决策记录

本文件记录需求讨论中已经明确确认的产品决策。只有标记为“已确认”的内容才能进入 PRD 和技术方案；尚未确认的内容不得在开发中自行补全。

## 已确认决策

### RD-001 产品形态

- 状态：已确认
- 决策：建设单商家、非收费、可公网部署的电商智能客服系统。
- 数据范围：使用模拟电商业务数据，不接入真实商家订单系统。
- 排除范围：不建设多商家 SaaS，不接入支付、充值、订阅和外部客服渠道。

### RD-002 消费者账号

- 状态：已确认
- 决策：系统必须提供消费者账号体系。
- 注册方式：消费者自行注册，不依赖管理员预先创建账号。
- 账号必须与业务用户身份绑定，登录后只能访问该账号对应的订单、物流、地址和售后数据。
- 公网环境不得允许消费者通过对话或请求参数切换为任意 `user_id`。

### RD-003 登录凭证

- 状态：已确认
- 决策：使用“邮箱 + 密码”作为消费者登录凭证。
- 注册时必须验证邮箱所有权。
- 系统必须支持登录、保持登录状态、退出登录、修改密码、忘记密码和邮件重置密码。
- 系统必须支持注销账号。
- 密码不得明文存储，邮箱验证码和密码重置凭证必须有有效期、单次使用和频率限制。

### RD-004 账号与模拟订单数据

- 状态：已确认
- 决策：消费者注册并完成邮箱验证后，系统为该账号初始化少量独立的模拟业务数据，使其登录后可以直接使用项目核心功能。
- 模拟数据保存在同一套业务表中，通过该账号绑定的 `user_id` 隔离；不是为每个账号复制数据库。
- 初始化数据只覆盖订单查询、物流查询、地址修改、订单取消和售后申请所需的代表性场景，不无限生成无关数据。
- 每个账号只能查询和修改绑定到自身 `user_id` 的模拟订单、物流、地址和售后记录。
- 不再设置与普通账号逻辑不同的共享演示账号；正常注册账号就是可完整体验项目功能的账号。
- 本版本不建设商城、商品购买、购物车和下单系统。

### RD-005 访问控制

- 状态：已确认
- 决策：消费者必须完成登录后才能进入客服并调用业务接口。
- 未登录用户只能访问注册、邮箱验证、登录、忘记密码和重置密码相关页面及接口。

### RD-006 模拟数据恢复

- 状态：已确认
- 决策：不向消费者提供“恢复初始数据”产品功能，避免把维护能力混入正常客服体验。
- 管理员可以针对指定账号重新初始化模拟业务数据，用于测试恢复和问题复现。
- 数据重置必须二次确认并记录审计日志。

### RD-007 管理员最小职责

- 状态：已确认
- 决策：提供系统管理员角色，但不建设商家管理后台。
- 管理员只负责查看和检索账号、启用或禁用账号、重置指定用户模拟数据、查看运行指标和审计记录。
- 管理员不得通过后台任意伪造真实交易，不提供商品、购物车、支付和商家配置功能。

### RD-008 Redis职责

- 状态：已确认
- 决策：本版本引入 Redis。
- MySQL保存账号、用户资料、订单、物流、地址、售后和审计等长期业务事实。
- Redis保存登录 Session、邮箱验证码、密码重置凭证、接口限流计数和对话 Tracker 等临时状态。
- Redis不得作为订单、物流和售后的唯一事实来源。

### RD-009 邮件发送

- 状态：已确认
- 决策：邮件模块使用标准 SMTP 接口，不在业务代码中绑定特定邮件厂商。
- SMTP主机、端口、账号、密码、发件人和TLS配置通过环境变量注入。
- 实际上线可配置支持SMTP的邮箱或事务邮件服务；更换供应商不得修改注册和密码重置业务逻辑。

### RD-010 账号注销

- 状态：已确认
- 决策：用户确认注销后，撤销全部登录 Session，并清除账号、用户资料及其关联的模拟订单、物流、地址和售后数据。
- 为保障指标一致性，历史运行日志只保留不可反查个人身份的匿名聚合数据。

### RD-011 初始容量目标

- 状态：已确认
- 决策：当前版本按单机部署和可验证的中低流量目标设计，不宣称未经验证的大规模承载能力。
- 初始容量目标为1000个注册账号、100个日活账号、20个并发客服会话。
- 月度可用性目标为99.5%，完整客服请求P95目标不超过15秒。
- 最终承载能力以压测和连续运行数据为准，达不到目标时必须记录真实结果并优化，不得修改统计口径。

## 尚未确认

当前产品方向没有阻断 PRD 重写的未确认项。具体云厂商、邮件供应商、域名和服务器规格属于部署选型，必须遵守上述产品边界，不得反向改变产品需求。
