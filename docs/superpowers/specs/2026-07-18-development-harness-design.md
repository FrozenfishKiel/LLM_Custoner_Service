# 电商智能客服系统开发 Harness 设计

## 1. 目标

本工作流用于把 PRD、技术方案、实现、测试和上线证据连接成可重复执行的工程闭环。它不改变产品范围，也不替代 `REQUIREMENTS_DECISIONS.md`、`PRD.md` 和 `TECHNICAL_DESIGN.md`。

核心原则：

- 每个实现任务必须关联 PRD 编号、验收条件和验证命令。
- 新行为和缺陷修复遵循测试先行，先观察预期失败，再完成最小实现。
- 开发自测与独立 QA 复测分离，不能用开发者自测代替验收。
- 正常用户旅程、异常边界、安全、压力、监控和恢复都属于完成定义。
- 目标值与实测值分栏记录，只有真实运行数据可以作为成果。

## 2. Agent 分工与并发上限

系统最多同时使用四个 Agent，包括主 Agent。

### 主 Agent

- 维护需求追踪、实施计划和依赖顺序。
- 拆分任务并判断是否可以并行。
- 审查所有 Agent 的修改和证据。
- 处理跨模块接口、迁移顺序和冲突。
- 运行集成后的全量质量门禁并维护最终验收状态。

### 开发 Agent

- 同一时间最多两个。
- 每个 Agent 只负责一个边界明确、文件改动不重叠的任务。
- 先提交失败测试证据，再实现最小改动并执行局部回归。
- 返回改动文件、验证命令、通过/失败数量、残余风险和待 QA 场景。
- 共享数据库结构、公共接口或相同文件的任务不得并行修改。

### 持续 QA Agent

- 从项目基线建立开始一直保留到上线验收结束。
- 可以维护测试代码、测试数据、故障注入、报告和测试基础设施。
- 不直接修改生产实现，缺陷必须回流给主 Agent 或对应开发 Agent。
- 每个可运行增量都独立复测，不等待所有功能开发完成。
- 报告失败时必须提供复现命令、预期、实际结果、影响范围和严重级别。

## 3. 增量开发闭环

每个任务按固定顺序执行：

```text
选定 PRD 编号和验收条件
  -> 写最小失败测试并确认失败原因
  -> 实现最小生产改动
  -> 开发 Agent 执行局部测试和相关回归
  -> 持续 QA Agent 独立复测
  -> 缺陷回流并补充回归测试
  -> 主 Agent 执行集成门禁
  -> 保存报告、指标和追踪矩阵状态
```

任何环节失败，任务保持未完成。修复缺陷时必须先加入能够稳定复现问题的回归测试。

## 4. 测试层级

### 单元测试

覆盖邮箱和密码规则、Token、Session、限流、身份映射、模拟数据幂等、Tracker 序列化、Flow 状态、Action 归属、事务和幂等判断。

### 集成测试

使用受控测试数据验证 MySQL、Redis、Neo4j 和 SMTP Adapter。集成测试必须具备隔离、清理和可重复执行能力，不依赖生产凭证。

### 浏览器端到端测试

模拟真实用户完成注册、验证、登录、刷新、退出、找回密码、聊天、五类客服流程、账号设置和注销。至少使用两个消费者账号验证数据隔离，并使用消费者和管理员账号互相验证权限边界。

### LLM 评测

版本化保存输入、对话历史、预期意图、槽位、Flow、Action 和最终状态。Prompt、模型、策略、Flow 或解析器变化后必须重跑。

### 压力与稳定性测试

验证 20 个并发客服会话、注册和登录突发流量、混合读写、长对话、连接池饱和和连续运行。记录吞吐、P50/P95/P99、错误率和资源峰值。

### 安全与风险测试

覆盖身份伪造、IDOR、管理员越权、CSRF、CORS、Session 重放、账号枚举、限流绕过、注入输入、敏感日志、依赖漏洞和镜像密钥扫描。

### 故障、监控与恢复测试

注入 DeepSeek、SMTP、MySQL、Redis 和 Neo4j 故障，验证稳定错误、事务回滚、告警触发与恢复关闭。实际执行应用重启、Redis 重启、MySQL/Neo4j 备份恢复和发布回滚。

## 5. 真实用户与刁钻场景

正常用户行为不只包括理想路径，还包括刷新、返回、重复点击、重复提交、改口、长时间停留、多个标签页、网络中断和重试。

边界场景至少包括：

- 空白、超长、Unicode、HTML/脚本、SQL/Cypher/Prompt 注入和损坏请求。
- 伪造 `sender_id`、`user_id`、`account_id`、Session 和对象 ID。
- 双重邮箱验证、并发初始化、同单并发取消、并发改地址和重复售后。
- 修改密码、禁用账号或注销时仍有在途请求。
- 模型超时、限流或返回畸形指令，数据库事务中断，依赖短暂不可用。
- 调试页、API 文档、Tracker、Domain、Flow、指标和管理接口的公网隔离。

## 6. 质量门禁

### 提交门禁

- 新增测试已先产生预期失败。
- 局部测试和相关回归全部通过。
- 不包含 `.env`、凭证、数据库卷、模型缓存或运行日志。
- 不扩大技术方案定义的文件改动范围。

### 模块门禁

- 对应 PRD 验收条件有自动化证据。
- 持续 QA Agent 独立复测通过。
- 异常、越权、重复提交和依赖故障行为明确。
- 新增日志和指标不泄露敏感信息。

### 发布门禁

- 单元、集成、E2E、LLM 评测、安全扫描和压测完成。
- 备份恢复、发布回滚和告警演练完成。
- HTTPS、内部接口隔离、密钥轮换和上线前置事项完成。
- 连续运行产生真实指标，所有未解决风险有明确结论。

## 7. 报告与量化数据

每次测试反馈使用统一字段：

```text
构建或提交标识
测试环境和依赖版本
测试范围与 PRD 编号
执行命令
通过、失败、跳过数量
失败复现与严重级别
性能和资源数据
新增或关闭的缺陷
残余风险
报告与证据路径
```

持续记录以下指标：

- 测试总数、通过率、覆盖率、回归缺陷数和缺陷修复时长。
- 注册、验证、登录、会话和各 Flow 的触发与完成率。
- 意图准确率、槽位 Precision/Recall/F1、Flow/Action 准确率、兜底率和越界率。
- 吞吐量、并发数、P50/P95/P99、4xx/5xx、超时率和资源峰值。
- DeepSeek、MySQL、Redis、Neo4j 和 SMTP 的成功率、延迟与恢复时间。
- 可用性、故障次数、MTTR、RTO、RPO、备份和回滚耗时。

PRD 初始目标为 20 个并发客服会话、非模型接口 P95 不超过 1 秒、完整客服请求 P95 不超过 15 秒、月可用性 99.5%。报告必须同时展示目标和实测值，不得把目标写成已经取得的结果。

## 8. 当前基线结论

首轮只读盘点表明当前自动化测试数量为零，且尚无 Git 安全基线、Redis、生产认证、监控、部署和恢复设施。公网化前的首要工作仍按交接顺序执行：Git 与配置安全，然后才进入数据库迁移和账号模型。
